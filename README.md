# RateUKMA — Kubernetes Infrastructure

GitOps infrastructure for [RateUKMA](https://github.com/ukma-cs-ssdm-2025/rate-ukma) — a course and professor rating application for NaUKMA students.

## Stack

| Component | Technology |
|---|---|
| Application | Django (Gunicorn) + React SPA (Vite) |
| Database | PostgreSQL — managed by **CloudNativePG** operator |
| Cache / Sessions | Redis (Deployment) |
| TLS | **cert-manager** + Let's Encrypt |
| GitOps | **FluxCD v2** |
| Secrets | **Sealed Secrets** (Bitnami) |
| Ingress | **NGINX Ingress Controller** |
| Packaging | **Helm** chart (`./charts/rate-ukma`) |

## Component Glossary

### k3s
Lightweight Kubernetes distribution. A single binary that bundles the Kubernetes control plane, kubelet, containerd, and a built-in load balancer (ServiceLB). Used instead of full kubeadm because it installs in under a minute and has ~512 MB overhead vs ~1 GB+ — practical for single-node cloud VMs. Traefik (the default k3s ingress) is disabled; we use NGINX Ingress instead.

### FluxCD v2
GitOps operator running inside the cluster. It watches this repository and reconciles the cluster state to match what is committed. Any change pushed to `main` is automatically applied — no `kubectl apply` needed. Consists of several controllers:
- **source-controller** — fetches and caches Git repos and Helm chart sources
- **kustomize-controller** — applies Kustomize manifests from the source
- **helm-controller** — installs and upgrades Helm releases declared as `HelmRelease` resources
- **image-reflector-controller** — scans container registries for new image tags
- **image-automation-controller** — commits updated image tags back to this repo when a new version is detected

### Kustomize
A Kubernetes-native configuration management tool built into `kubectl`. Used here to compose and layer YAML manifests — for example, `infrastructure/controllers/kustomization.yaml` lists all operator subdirectories so Flux applies them as a group. No templating; overlays are applied by patching base resources.

### Helm
Kubernetes package manager. The app is packaged as a Helm chart (`charts/rate-ukma/`) which templates all Deployments, Services, Ingress, HPA, etc. Environment-specific values (image tags, replica counts, resource limits) are set in `environments/*/helmrelease.yaml` without touching the templates.

### CloudNativePG (CNPG)
Kubernetes operator for managing PostgreSQL clusters. Instead of running a plain Postgres container, CNPG manages the full lifecycle: primary election, streaming replication, automated failover, and backup. Declared as a `Cluster` CR in the Helm chart. Production runs a primary + 1 read replica; staging runs a single instance.

### cert-manager
Kubernetes operator for automating TLS certificate management. Watches `Ingress` resources annotated with `cert-manager.io/cluster-issuer`, requests certificates from Let's Encrypt via the ACME HTTP-01 challenge, and stores them as Kubernetes `Secret` objects. Renews certificates automatically before expiry. Runs as three pods: the core controller, a CA injector, and an admission webhook.

### Sealed Secrets
A controller (by Bitnami) that allows encrypted secrets to be safely committed to Git. The `kubeseal` CLI encrypts a plain Kubernetes `Secret` using the cluster's public key into a `SealedSecret` resource. Only the controller running in that specific cluster can decrypt it. Secrets sealed for production cannot be decrypted by staging and vice versa.

### NGINX Ingress Controller
Routes external HTTP/HTTPS traffic into the cluster. A single NGINX pod handles all ingress — TLS termination, routing by hostname and path, and forwarding to backend Services. k3s's ServiceLB assigns the node's external IP to the LoadBalancer Service that exposes NGINX on ports 80 and 443.

### HPA (Horizontal Pod Autoscaler)
A native Kubernetes resource (no extra install needed) that automatically scales the number of backend pod replicas based on CPU utilization. Configured in production to maintain between 2 and 5 replicas, targeting 70% CPU usage. Powered by the `metrics-server` which is bundled with k3s.

### Frontend pod (init container pattern)

The frontend Deployment uses two init containers before the nginx container starts:

1. **copy-webapp** — copies `/dist` from the webapp image into a shared `emptyDir` volume
2. The nginx container serves static files from that volume and reverse-proxies `/api/`, `/admin/`, `/accounts/`, `/courses/` to the backend Service

This avoids needing a custom nginx image — the webapp image ships its own built assets.

### Migrations

Django migrations run as a Kubernetes init container on the backend Deployment, before Gunicorn starts. This ensures migrations are applied on every deploy before traffic hits the new pods.

## Day-to-day Operations

### Deploying a new version

**You don't need to touch this repo for a normal release.** Push a semver tag to the app repo and CI does the rest:

```
git tag v1.2.3 && git push origin v1.2.3
```

CI builds and pushes:
- `ghcr.io/.../backend:v1.2.3`
- `ghcr.io/.../webapp:k8s-beta-v1.2.3`

Flux Image Automation detects the new tags within ~1 minute, commits the updated tags to `environments/production/helmrelease.yaml`, and Flux performs a rolling update.

### Applying infrastructure changes

Edit the relevant files and push to `main`. Flux reconciles every 5–10 minutes, or force it immediately:

```bash
# Force reconcile everything
KUBECONFIG=~/.kube/prod.yaml flux reconcile kustomization flux-system --with-source

# Force reconcile only the app
KUBECONFIG=~/.kube/prod.yaml flux reconcile helmrelease rate-ukma -n production

# Force reconcile only infrastructure
KUBECONFIG=~/.kube/prod.yaml flux reconcile kustomization infrastructure
```

### Checking cluster status

```bash
# Overall Flux status
KUBECONFIG=~/.kube/prod.yaml flux get all -A

# HelmRelease status
KUBECONFIG=~/.kube/prod.yaml flux get helmreleases -A

# Image automation (what tags Flux sees)
KUBECONFIG=~/.kube/prod.yaml flux get image policy -A

# Pod status
KUBECONFIG=~/.kube/prod.yaml kubectl get pods -n production

# HPA (autoscaler)
KUBECONFIG=~/.kube/prod.yaml kubectl get hpa -n production

# TLS certificate
KUBECONFIG=~/.kube/prod.yaml kubectl get certificate -n production
```

### Reading logs

```bash
# Backend logs
KUBECONFIG=~/.kube/prod.yaml kubectl logs -n production deploy/production-rate-ukma-backend

# Frontend (nginx) logs
KUBECONFIG=~/.kube/prod.yaml kubectl logs -n production deploy/production-rate-ukma-frontend

# Follow logs in real time
KUBECONFIG=~/.kube/prod.yaml kubectl logs -n production deploy/production-rate-ukma-backend -f
```

### Updating a secret

Sealed Secrets are encrypted per-cluster — the same plaintext secret must be sealed separately for staging and production.

```bash
# Edit the plaintext secret file (never commit these)
# secret-django.yaml / secret-postgres.yaml

# Re-seal for production
kubeseal --cert prod-cert.pem --format yaml \
  < secret-django.yaml \
  > environments/production/sealed-secrets/django-sealed.yaml

# Commit and push — Flux applies the new secret automatically
```

If you don't have the cert locally:

```bash
KUBECONFIG=~/.kube/prod.yaml kubeseal --fetch-cert \
  --controller-namespace kube-system \
  --controller-name sealed-secrets-controller \
  > prod-cert.pem
```

### Modifying Helm chart values

Environment-specific values live in `environments/production/helmrelease.yaml`. Edit and push:

```yaml
# Example: scale up backend replicas
backend:
  replicaCount: 3
```

Flux reconciles within 5 minutes. To apply immediately:

```bash
KUBECONFIG=~/.kube/prod.yaml flux reconcile helmrelease rate-ukma -n production
```

### Changing Helm chart templates

Edit files under `charts/rate-ukma/templates/`. After pushing, Flux may need a nudge to re-render the chart (the chart version doesn't change automatically):

```bash
KUBECONFIG=~/.kube/prod.yaml kubectl annotate helmchart.source.toolkit.fluxcd.io production-rate-ukma \
  -n flux-system reconcile.fluxcd.io/requestedAt="$(date -u +%Y-%m-%dT%H:%M:%SZ)" --overwrite

KUBECONFIG=~/.kube/prod.yaml kubectl annotate helmrelease rate-ukma -n production \
  reconcile.fluxcd.io/forceAt="$(date -u +%Y-%m-%dT%H:%M:%SZ)" --overwrite
```

## Bootstrap (from scratch)

Only needed when provisioning a new cluster. The production cluster is already running.

### 1. Provision VM and install k3s

```bash
# Run on the VM
curl -sfL https://get.k3s.io | sh -s - --disable traefik

# Copy kubeconfig to local machine
scp root@<vm-ip>:/etc/rancher/k3s/k3s.yaml ~/.kube/prod.yaml
sed -i 's/127.0.0.1/<vm-ip>/' ~/.kube/prod.yaml
```

### 2. Install local tools

```bash
# Flux CLI
curl -s https://fluxcd.io/install.sh | bash

# kubeseal (Linux)
KUBESEAL_VERSION=$(curl -s https://api.github.com/repos/bitnami-labs/sealed-secrets/releases/latest \
  | grep tag_name | cut -d '"' -f4 | sed 's/^v//')
curl -Lo kubeseal.tar.gz \
  "https://github.com/bitnami-labs/sealed-secrets/releases/download/v${KUBESEAL_VERSION}/kubeseal-${KUBESEAL_VERSION}-linux-amd64.tar.gz"
tar -xzf kubeseal.tar.gz kubeseal && sudo install -m 755 kubeseal /usr/local/bin/kubeseal
```

### 3. Bootstrap Flux

```bash
export GITHUB_TOKEN=<pat-with-repo-write-access>

KUBECONFIG=~/.kube/prod.yaml flux bootstrap github \
  --owner=stasiaaleks \
  --repository=rate-ukma-deployment \
  --branch=main \
  --path=clusters/production \
  --personal \
  --components-extra=image-reflector-controller,image-automation-controller
```

Flux commits bootstrap files into `clusters/production/flux-system/`. Pull before continuing:

```bash
git pull
```

### 4. Create the GHCR pull secret

The app images are in a private GHCR package. Create an image pull secret in the app namespace:

```bash
KUBECONFIG=~/.kube/prod.yaml kubectl create namespace production

KUBECONFIG=~/.kube/prod.yaml kubectl create secret docker-registry ghcr-auth \
  --docker-server=ghcr.io \
  --docker-username=<github-username> \
  --docker-password=<github-pat-with-read-packages> \
  -n production
```

### 5. Grant Flux write access to this repo

Flux Image Automation commits tag updates back to this repo. The deploy key created by `flux bootstrap` must have **write access**:

1. Go to **GitHub → `rate-ukma-deployment` → Settings → Deploy keys**
2. Delete the read-only key Flux created
3. Re-add the same public key with **Allow write access** checked

Get the public key:

```bash
KUBECONFIG=~/.kube/prod.yaml kubectl -n flux-system get secret flux-system \
  -o jsonpath='{.data.identity\.pub}' | base64 -d
```

### 6. Seal and commit secrets

Each cluster has its own Sealed Secrets controller with its own encryption key. Secrets sealed for production **cannot** be decrypted by staging and vice versa.

```bash
# Fetch the cluster's public cert (wait for sealed-secrets-controller to be Ready first)
KUBECONFIG=~/.kube/prod.yaml kubeseal --fetch-cert \
  --controller-namespace kube-system \
  --controller-name sealed-secrets-controller \
  > prod-cert.pem
```

Create plaintext secret files (do **not** commit these — they are in `.gitignore`):

```yaml
# secret-django.yaml
apiVersion: v1
kind: Secret
metadata:
  name: rate-ukma-django
  namespace: production
type: Opaque
stringData:
  DJANGO_SECRET_KEY: "<python -c 'from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())'>"
  MICROSOFT_CLIENT_ID: "<azure-app-client-id>"
  MICROSOFT_CLIENT_SECRET: "<azure-app-client-secret>"
  MICROSOFT_TENANT_ID: "<azure-tenant-id>"
  SENTRY_DSN_BACKEND: ""
  CORPORATE_EMAIL: ""
  CORPORATE_PASSWORD: ""
```

```yaml
# secret-postgres.yaml
apiVersion: v1
kind: Secret
metadata:
  name: rate-ukma-postgres
  namespace: production
type: Opaque
stringData:
  username: rateukma
  password: "<strong-random-password>"
```

Seal and commit:

```bash
kubeseal --cert prod-cert.pem --format yaml \
  < secret-django.yaml \
  > environments/production/sealed-secrets/django-sealed.yaml

kubeseal --cert prod-cert.pem --format yaml \
  < secret-postgres.yaml \
  > environments/production/sealed-secrets/postgres-sealed.yaml

git add environments/production/sealed-secrets/
git commit -m "feat: add production sealed secrets"
git push
```

### 7. Update Azure App Registration

Add the callback redirect URI in **Azure Portal → App registrations → Authentication**:

```
https://k8s-beta.rateukma.com/accounts/microsoft/login/callback/
```

```bash

# All pods should be Running
```

## Image Automation

Flux polls GHCR every minute and tracks these tag patterns:

| Policy | Pattern | Tracks |
|---|---|---|
| `rate-ukma-backend` | `v*.*.*` (semver) | Backend image for all envs |
| `rate-ukma-webapp-k8s-beta` | `k8s-beta-v*.*.*` | Frontend for production |
| `rate-ukma-webapp-staging` | `staging-v*.*.*` | Frontend for staging |
| `rate-ukma-webapp-live` | `live-v*.*.*` | (legacy live env) |

When a new tag is detected, `ImageUpdateAutomation` commits the updated tag back to `environments/*/helmrelease.yaml` and Flux reconciles.

The `# {"$imagepolicy": "..."}` comments on `tag:` lines in `helmrelease.yaml` are markers that tell Flux where to write the tag.

## Troubleshooting

**HelmRelease not updating after chart template change**

Flux only re-renders when the chart version or values change. Force a re-render:

```bash
KUBECONFIG=~/.kube/prod.yaml kubectl annotate helmchart.source.toolkit.fluxcd.io production-rate-ukma \
  -n flux-system reconcile.fluxcd.io/requestedAt="$(date -u +%Y-%m-%dT%H:%M:%SZ)" --overwrite

KUBECONFIG=~/.kube/prod.yaml kubectl annotate helmrelease rate-ukma -n production \
  reconcile.fluxcd.io/forceAt="$(date -u +%Y-%m-%dT%H:%M:%SZ)" --overwrite
```

**New infrastructure CRD race (e.g. adding a new operator)**

If a new operator's CRD is applied in the same Kustomization as a CR that uses it, Flux will fail with "no matches for kind". Fix: apply the operator first (remove the CR from the kustomization, push, wait for operator to be Ready, then add the CR back).

**ImagePullBackOff**

The `ghcr-auth` secret must exist in the app namespace. Re-create it if missing (see step 4 of bootstrap).

**Flux Image Automation not committing**

The deploy key must have write access to this repo (see step 5 of bootstrap). Check:

```bash
KUBECONFIG=~/.kube/prod.yaml flux get image update rate-ukma -n flux-system
```
